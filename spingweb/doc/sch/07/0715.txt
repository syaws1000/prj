## 🌟 웹프로젝트 UML 유즈케이스 다이그램 작성 교안 (학습자 공유용) 🌟

---

### 1. 🧠 주의 집중 끌기 및 학습 연결성 제공 (Gain Attention / Relevance)

* **질문 던지기**:
  “자, 여러분은 카페에서 ‘주문’할 때 시스템이 어떤 과정을 거칠지 생각해 본 적 있나요?”
* **짧은 영상 or 짧은 연기극**

  > 예: “개발자가 없는 카페 주문 시스템의 혼란 상황 연기”
* **흥미 요소 삽입**
  “이 수업이 끝나면 여러분은 백종원의 카페 프랜차이즈 시스템도 설계할 수 있습니다! (…은 아니고, 그 절반쯤까지는 😄)”

---

### 2. 🎯 학습 목표 (Inform Learners of Objectives / Confidence)

* UML 유즈케이스 다이어그램의 정의와 목적을 설명할 수 있다.
* 웹 프로젝트 시나리오를 분석하여 액터와 유즈케이스를 도출할 수 있다.
* StarUML 또는 Lucidchart를 활용하여 유즈케이스 다이어그램을 직접 설계할 수 있다.
* 실무 환경에서 유즈케이스 다이어그램을 활용하여 요구사항을 정리할 수 있다.

---

### 3. 🔍 선수 학습 상기 및 실생활 연결하기 (Stimulate Recall / Relevance)

* **사전지식 연결**

  > “요구사항 분석, 기억나나요? 우리가 만든 그 쇼핑몰 시나리오 기억나죠?”
* **실생활 예시**

  > “배달의 민족에서 음식 주문할 때 어떤 행위들이 필요한지 떠올려 봅시다.”
* **그림 퀴즈**

  > PPT로 시스템 구성 요소 맞추기 (“누가? 무엇을? 언제?”)

---

### 4. 🧠 주요 개념/핵심 키워드 - 최고의 암기법 제시 (Present Content / Attention)

* **핵심 키워드와 암기법**

  | 개념              | 설명                    | 암기법(유머 포함)             |
  | --------------- | --------------------- | ---------------------- |
  | Actor           | 시스템 외부 사용자            | "밖에서 문 두드리는 사람"        |
  | Use Case        | 시스템이 제공하는 기능          | "뭘 해줄까? 시스템의 서비스 메뉴"   |
  | System Boundary | 시스템의 경계               | "놀러올 수는 있어도 넘어오면 안 돼!" |
  | Association     | Actor와 Use Case 간의 관계 | “둘이 손잡은 선 – 연애는 아님 😅” |

---

### 5. 🧩 학습 목표별 탐구형 학습 (Provide Learning Guidance / Confidence)

* **탐구 과제 제시**
  “여러분이 자주 사용하는 웹사이트(배민, 인스타, 넷플릭스 등)를 떠올려 보세요.
  ‘사용자’는 누구고, ‘기능’은 무엇일까요?”
* **가이드 질문**

  * 사용자는 어떤 요청을 할까?
  * 시스템은 어떤 반응을 보여줘야 할까?
  * 기능은 내부 기능일까 외부에서 보이는 기능일까?
* **힌트 제공**

  > "주문을 요청하는 고객은 **Actor**, 주문을 처리하는 기능은 **Use Case**"

---

### 6. 🛠 실습 활동 및 협력 학습 유도 (Elicit Performance / Satisfaction)

* **실습1**: StarUML을 활용한 유즈케이스 다이어그램 실습

  * 시나리오: “고객이 도서 쇼핑몰에서 책을 검색하고 주문한다.”
  * 구성요소: 고객, 관리자, 주문하기, 장바구니 추가 등
* **실습2**: 팀별 실무 시나리오 설계

  * 주제 예시: "헬스장 등록 시스템", "학원 예약 시스템", "카페 주문 시스템"
  * 팀원끼리 액터/기능 도출 후 발표

---

### 7. ✅ 단계별 피드백 제공과 자기 평가 기회 (Provide Feedback / Confidence)

* **중간 점검 체크리스트**

  * [ ] 액터는 정확히 식별되었는가?
  * [ ] 기능은 하나의 Use Case로 표현되었는가?
  * [ ] 연관 관계가 선명히 드러나는가?
* **자기 평가 카드**

  * 💬 “오늘 나는 시스템 분석가가 된 느낌이었다.”
  * 😵 “다이어그램 그리다 머리가 꼬였다.”
  * 😊 “다음엔 더 복잡한 시스템에도 도전할 수 있을 듯!”

---

### 8. 📊 수행 평가 및 피드백 제공 (Assess Performance / Satisfaction)

* **수행 평가 기준**

  | 항목         | 평가내용         | 배점  |
  | ---------- | ------------ | --- |
  | 액터 도출의 적절성 | 실사용자 정확히 도출  | 20점 |
  | 유즈케이스 구성   | 핵심 기능 분리 표현  | 30점 |
  | 다이어그램 구조   | 시각적 명확성과 일관성 | 30점 |
  | 팀별 발표 및 설명 | 논리적 흐름 설명력   | 20점 |
* **피드백 방식**

  * 동료 피드백 (좋았던 점, 아쉬운 점, 궁금한 점)
  * 교사 피드백 (개별 + 그룹)

---

### 9. 🧩 학습 전이 강화 및 실제 문제 적용 활동 (Enhance Retention and Transfer / Relevance)

* **응용 과제**
  “여러분이 원하는 스타트업 웹서비스를 구상하고, 초기 기획서를 위한 유즈케이스 다이어그램을 만들어 보세요.”
